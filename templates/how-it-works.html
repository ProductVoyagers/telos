Agents & Registry

Four specialist subagents do the judgment work in isolation. The registry is a deterministic script — judgment goes to the models, mechanics go to code.

🎨

Render Agent

agents/telos-render.md

Two modes: Create (wireframe to screen) or Modify (apply changes to existing screen). Reads design system from design-system.json. Outputs versioned files (-v1, -v2).

Input
Wireframe image or existing screen + modifications
Output
HTML file at projects/.../-v1.html
🎨

Design System Agent

agents/telos-design-system.md

Analyzes 3-5 product screenshots and extracts design tokens (colors, typography, radius, components) into design-system.json.

Input
3-5 product screenshots
Output
design-system.json
🔍

Critique Agent

agents/telos-critique.md

Evaluates flow screens against KRs. Produces a critique card with alignment score, strengths, gaps, and suggestions.

Input
Screen file paths, KR statement(s)
Output
critique-card.html
📊

Eval Agent

agents/telos-eval.md

Scores critique quality using 6 dimensions. Produces confidence badge data. Recommends retry if below threshold.

Input
Critique content, KR, mode (single/suite)
Output
Score X/30, per-dimension breakdown
📁

Registry (script, not an agent)

bin/telos-registry · python3

Deterministic bookkeeping — never a language model touching your files or repo. Updates manifest.json, generates the walkthrough, tracks versions, checks flow ownership for squads. In local mode it just writes files; connect a repo and it pulls, commits, and pushes with squad-safe conflict merging.

Input
Screen path, flow/project names, version info
Output
Updated manifest.json + generated walkthrough; git sync when connected
